## WCAG 2.2 criteria — new in 2.2

### 2.4.11 Focus Appearance — Minimum (AA)

**Requirement**: Focus indicator area ≥ perimeter × 1px; contrast ≥ 3:1.

**Implementation**:
- All interactive elements use Tailwind's `focus-visible:ring-2` (2px solid ring).
- `--ring` CSS variable changed from `oklch(0.708 0 0)` (~2:1 on white) to `oklch(0.145 0 0)` (~14:1 on white) — see [`app/globals.css`](app/globals.css).
- Global `@layer base { :focus-visible { --tw-ring-offset-width: 2px; --tw-ring-offset-color: var(--background); } }` ensures a white perimeter between element and ring, so the near-black ring contrasts against both dark-background buttons and the light page.

### 2.4.12 Focus Appearance — Enhanced (AAA)

**Requirement**: Focus indicator area ≥ perimeter × 2px; contrast ≥ 4.5:1.

**Implementation**: Same ring-2 (2px) ring already satisfies the area requirement (area = perimeter × 2px). The upgraded `--ring` color gives ~14:1 contrast — well above 4.5:1.

### 3.3.7 Redundant Entry (A)

**Requirement**: Information entered previously is auto-populated or not required again.

**Implementation**: The sign-in form uses `autoComplete="email"` and `autoComplete="current-password"`. The demo-account fill buttons populate both fields at once, eliminating re-entry.

### 3.3.8 Accessible Authentication — Minimum (AA)

**Requirement**: No cognitive function test is required unless an alternative is provided.

**Implementation**: Password authentication is a cognitive function test. The sign-in page provides "fill" buttons for each demo account, allowing users to authenticate without recalling credentials. This satisfies the exception in SC 3.3.8.

### 3.3.9 Accessible Authentication — Enhanced (AAA)

**Requirement**: Authentication does not rely on a cognitive function test.

**Implementation**: The demo-account fill buttons eliminate any need to recall or transcribe credentials — one click fills and a second click submits. Combined with browser `autoComplete`, authentication can complete with zero cognitive recall.

## WCAG 2.1 criteria

### 1.3.1 Info and Relationships (A)

- `<header>`, `<main>`, `<aside>`, `<nav>`, `<section>`, `<figure>`, `<blockquote>` used throughout.
- Tables in home page have `<thead>`, `<th scope>`.
- Lists that CSS would reformat use `role="list"` to preserve semantics.

### 1.3.5 Identify Input Purpose (AA)

- Email input: `autoComplete="email"` — [`app/auth/signin/page.tsx`](app/auth/signin/page.tsx:56)
- Password input: `autoComplete="current-password"` — [`app/auth/signin/page.tsx`](app/auth/signin/page.tsx:70)

### 2.1.1 Keyboard (A)

- All interactive elements are natively focusable (links, buttons, inputs).
- Drag-and-drop section reordering supports keyboard via dnd-kit `KeyboardSensor` with `sortableKeyboardCoordinates` — [`components/studio/SectionList.tsx`](components/studio/SectionList.tsx:109).
- The "Add section" dropdown (custom ARIA listbox) implements full keyboard control: `↓`/`↑` navigate options, `Home`/`End` jump to ends, `Enter`/`Space` select, `Escape` closes and returns focus to trigger — [`components/studio/StudioEditor.tsx`](components/studio/StudioEditor.tsx).

### 2.1.2 No Keyboard Trap (A)

- The "Add section" dropdown returns focus to the trigger on `Escape` or `Tab`.
- Base UI `Dialog` manages focus trap and restoration natively.

### 2.4.1 Bypass Blocks (A)

- A "Skip to main content" link is the first focusable element in the body. It is visually hidden (`.sr-only`) until focused, then appears at top-left — [`app/layout.tsx`](app/layout.tsx).
- Every page exposes `id="main-content"` on its `<main>` element as the skip target.

### 2.4.3 Focus Order (A)

- DOM order mirrors visual order throughout. No `tabIndex > 0` is used.

### 2.4.6 Headings and Labels (AA)

- Each page has exactly one `<h1>`.
- Studio: visually-hidden `<h1>Page Studio — editor</h1>` in the sidebar `<header>`; the page name visible in the sidebar is a `<p>` element, preventing a duplicate `<h1>` when the preview pane renders a Hero section with its own `<h1>` — [`components/studio/StudioEditor.tsx`](components/studio/StudioEditor.tsx).
- Heading hierarchy per page:
  - Home: `h1` "Pages" → `h2` per page card title
  - Sign-in: `h1` "Page Studio"
  - Preview: `h1` from Hero section → `h2` from CTA/FeatureGrid → `h3` from feature items
  - Studio: `h1` (sr-only) "Page Studio — editor" → `h2` "Properties"

### 2.4.7 Focus Visible (AA)

Implemented via `focus-visible:` Tailwind utilities on every interactive element. The `:focus-visible` pseudo-class means focus rings only appear for keyboard navigation, not pointer clicks.

### 2.4.9 Link Purpose — Link Only (AAA)

- "Preview" and "Edit" links in the page list include an `<span className="sr-only">` with the page title, making the accessible name unique per link (e.g. "Preview Home", "Edit Pricing") — [`app/page.tsx`](app/page.tsx).

### 2.4.10 Section Headings (AAA)

- Every content section uses a heading or `aria-labelledby`:
  - `Hero`: `<section aria-labelledby="hero-heading">` + `<h1 id="hero-heading">`
  - `CTA`: `<section aria-labelledby="cta-heading">` + `<h2 id="cta-heading">`
  - `FeatureGrid`: conditional `aria-labelledby="feature-grid-heading"` + `<h2>`
  - `Testimonial`: `<figure>` + `<figcaption>`

### 3.3.1 Error Identification (A)

- Form fields in PropsForm validate on blur. Invalid fields get `aria-invalid="true"`, `aria-describedby` pointing to an error `<p role="alert" id="...">` — [`components/studio/PropsForm.tsx`](components/studio/PropsForm.tsx).
- Sign-in errors use `<p role="alert">`.

### 3.3.2 Labels or Instructions (A)

- All inputs have explicit `<label htmlFor="...">` associations.
- Required fields display `aria-required="true"` and a visually marked asterisk (`<span aria-hidden="true">*</span>`).

### 4.1.2 Name, Role, Value (A)

- Custom ARIA listbox in "Add section" uses `role="listbox"`, `role="option"`, `aria-selected`, `aria-expanded`, `aria-haspopup="listbox"`, `aria-controls`, `aria-activedescendant`.
- Draggable items use `aria-label` for the drag handle and remove button.
- Publish/error feedback regions use `role="status" aria-live="polite"` and `role="alert"`.

### 4.1.3 Status Messages (AA)

- Publish success: `<div role="status" aria-live="polite">` announces version and changelog.
- Publish error: `<div role="alert">` announces immediately.
- Loading state: `aria-busy="true"` on the loading container.

## Prefers-reduced-motion

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    animation-iteration-count: 1 !important;
    transition-duration: 0.01ms !important;
    scroll-behavior: auto !important;
  }
}
```

All CSS transitions and animations are suppressed when the OS preference is set — [`app/globals.css`](app/globals.css).

## Known limitations

| Area | Status | Notes |
|------|--------|-------|
| Colour contrast (1.4.6 AAA, 7:1) | Not fully verified | Colours are achromatic oklch; light mode foreground `oklch(0.145)` on white background yields ~14:1 for body text. Muted-foreground `oklch(0.556)` on white is ~5.3:1 (meets AA, below AAA 7:1). |
| `prefers-color-scheme` dark | Structural only | Dark mode colours available via `.dark` class; automatic OS-level toggle not wired. |
| Mobile touch focus | Not tested | `focus-visible` is keyboard-only by spec; pointer devices do not show rings. |
